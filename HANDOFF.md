# Handoff: mcp-hayabusa

## What we built

An MCP server (`server.py`, built on the `mcp` library's `FastMCP`) with two layers. The first (this section) is Hayabusa EVTX scanning: two tools, `scan_evtx`, which shells out to the Hayabusa CLI to analyze Windows EVTX files and returns findings as structured JSON, and `get_hayabusa_rules`, which lists/filters the local Sigma/Hayabusa rule set so a caller can see what detections exist before scanning. The second layer, a detection-engineering knowledge base built on top of the same server, is covered in its own section below ("Detection engineering knowledge base") with its own What we built/How to use it/What's left/Decisions breakdown.

- **Tool**: `scan_evtx(file_path: str, min_severity: str | None = None, rule_filter: str | None = None, output_format: str = "summary", max_results: int | None = None) -> dict`
  - Validates the EVTX file exists before doing anything else.
  - Validates `min_severity` against Hayabusa's five levels: `informational`, `low`, `medium`, `high`, `critical`. Validates `output_format` is `summary` or `full`, and `max_results` is `>= 0` if given.
  - Locates the Hayabusa binary by globbing `./hayabusa/` for `hayabusa*.exe` (Windows) or `hayabusa*` (other platforms) — handles the real, version-suffixed filename (e.g. `hayabusa-3.10.0-win-x64.exe`) rather than assuming a literal `hayabusa.exe`.
  - Runs `hayabusa json-timeline -f <file> -o <tmp>.jsonl -L -w -q -Q`, optionally adding `-m <min_severity>` to filter at the source using Hayabusa's own `--min-level` rule filter (no separate Python-side filtering logic).
  - Parses the JSONL output (one JSON object per line), then applies `rule_filter` (case-insensitive substring match on `RuleTitle`), `output_format` (trims to `SUMMARY_FIELDS` when `"summary"`, the default), and `max_results` (truncates the list) — all client-side, since Hayabusa has no equivalent flags.
  - Returns `{"file", "min_severity", "rule_filter", "output_format", "count", "returned", "truncated", "findings"}` on success (`count` = total after `rule_filter`; `returned` = count after `max_results`; `truncated` = whether the cap bit), or `{"error": ...}` (with `stderr`/`returncode` where relevant) on any failure: missing file, invalid severity/output_format/max_results, missing binary, non-zero exit, timeout (5 min cap), or malformed output.
- **Tool**: `get_hayabusa_rules(keyword: str | None = None, max_results: int | None = 50) -> dict`
  - Walks `./hayabusa/rules/**/*.yml` (~5,000 files across `hayabusa/` and `sigma/`), skipping `.git`.
  - Extracts `title`, `id`, `level`, `status`, `ruletype`, `tags`, `description` per rule via `_parse_rule_summary()`, a regex line-scan over the rule file's unindented top-level fields (title/id/level/status/tags/description can appear before *or* after the indented `detection:` block — see the bug note under Decisions) — not a full YAML parser.
  - Filters to rules where `keyword` (case-insensitive) appears in the joined title/description/tags, if given.
  - Returns `{"keyword", "count", "returned", "truncated", "rules"}`, capped at `max_results` (default 50), or `{"error": ...}` if the rules directory is missing or `max_results` is invalid.
- **`scripts/download_hayabusa.py`**: downloads the latest Hayabusa release for the current OS/architecture from the GitHub API (no hardcoded version or URL — discovered dynamically), picks the right asset (excludes `-live-response` variants, prefers glibc over musl on Linux), and extracts it to `./hayabusa/`.
- **`scripts/download_sample_evtx.py`**: downloads one real attack sample (`Credential Access/4794_DSRM_password_change_t1098.evtx`, ~68KB, from [EVTX-ATTACK-SAMPLES](https://github.com/sbousseaden/EVTX-ATTACK-SAMPLES)) into `samples/` for manual testing.
- **`tests/test_scan_evtx.py`**: a simple manual test script (not pytest) that imports `server.scan_evtx` and `server.get_hayabusa_rules` directly and exercises each tool's parameters and error paths against the downloaded sample/rule set.
- **`.mcp.json`** (gitignored): registers this server with Claude Code locally, launched as `python server.py` with `cwd` pinned to this project's absolute path. Kept out of git because the `cwd` is machine-specific. (Not `.claude/settings.local.json` — that file is for permissions/hooks/env; Claude Code doesn't read MCP server definitions from it.)
- **Claude Desktop registration**: also registered directly in Claude Desktop's `claude_desktop_config.json` (separately from `.mcp.json` — Desktop and Code don't share config). See the Decisions section below for why that file's location and entry shape are both non-obvious on this machine.
- **`requirements.txt`**: just `mcp` — everything else needed (subprocess, JSON parsing) is standard library.

## What's left to do

- **No automated test framework.** `tests/test_scan_evtx.py` is a manual script, not `pytest`/`unittest` — no CI; assertions now back the newer checks, but there's still an early `sys.exit(1)` pattern for the setup/precondition checks rather than a real test runner.
- **No handling for multi-file/directory scans.** Hayabusa's `-d/--directory` input mode isn't exposed; `scan_evtx` only supports a single file via `-f`.
- **`rule_filter` is a client-side substring match on `RuleTitle`, not a real Hayabusa rule selector.** Hayabusa's own rule-selection flags (`-r`, `-c`, `--include-tag`, `--include-category`, etc.) still aren't surfaced — `rule_filter` only filters findings *after* Hayabusa has already run every rule, it doesn't change which rules execute. For a large EVTX file this means no scan-time speedup, just a smaller result set.
- **No `update-rules` integration.** Hayabusa ships with a `update-rules` subcommand to pull the latest detection rules; nothing currently calls it, so rules are whatever `download_hayabusa.py`'s release zip bundled. `get_hayabusa_rules` reflects whatever's on disk, so it's only as current as the last `download_hayabusa.py` run.
- **`get_hayabusa_rules`'s YAML field extraction is a regex line-scan, not a real parser.** Works for the current rule set's consistent unindented-top-level-field style (see the fixed bug under Decisions), but a rule that nests a scalar field unusually (e.g. inside a `related:`/`references:` sub-mapping using an unindented continuation) could still parse incorrectly. Low risk since it's a discovery/listing aid, not used for actual scanning.

- **No shared MCP registration.** `.mcp.json` is gitignored (absolute-path `cwd`, machine-specific). If the team wants this server auto-registered for every collaborator, a portable (non-absolute-path) shared `.mcp.json` would need to be worked out; right now each collaborator creates their own.

## Decisions made and why

- **`FastMCP` over the low-level `mcp.server` API.** Simpler decorator-based tool registration (`@mcp.tool()`), and it was already the documented high-level entry point in the installed `mcp==1.28.1` package.
- **JSONL output (`-L`), not the default `-o` JSON format.** Initially implemented against Hayabusa's `-o results.json` on the assumption it was a single JSON array (per `--help`: "Save the timeline in JSON format"). Testing against a real EVTX file showed the actual output is pretty-printed JSON objects concatenated back-to-back with no array wrapper or separators — not valid JSON, not JSONL. Switched to `-L`/JSONL output, which is one compact object per line and parses trivially with `json.loads` per line.
- **Severity filtering delegated to Hayabusa's own `-m/--min-level` flag**, rather than fetching everything and filtering in Python. Hayabusa's five levels map exactly to what was asked for, so there was no reason to duplicate that logic client-side.
- **Binary lookup globs for `hayabusa*` instead of assuming a fixed filename.** The user's assumption was `./hayabusa/hayabusa` or `hayabusa.exe`, but `download_hayabusa.py` extracts the release zip's actual filename, which is version-suffixed (`hayabusa-3.10.0-win-x64.exe`). The glob reconciles the two without requiring a rename step.
- **`download_hayabusa.py` and `download_sample_evtx.py` use only the GitHub API + stdlib (`urllib`, `zipfile`, `json`) — no `requests` dependency**, and asset/file selection is done dynamically (via the releases API and a repo tree listing) rather than hardcoding release URLs, so the scripts don't silently break when Hayabusa cuts a new version.
- **`.mcp.json`, not `.claude/settings.json`/`settings.local.json`, for the MCP server registration.** Claude Code only reads `mcpServers` from `.mcp.json` (project root) or `~/.claude.json` (via `claude mcp add`) — `settings.json`/`settings.local.json` are for permissions/hooks/env and are silently ignored for this purpose. `.mcp.json` also hardcodes an absolute Windows path (`cwd`) specific to this machine, so it's gitignored rather than committed; a collaborator on another machine/OS creates their own.
- **5-minute subprocess timeout on the Hayabusa scan.** Defensive against a hung or unexpectedly slow scan blocking the MCP server indefinitely; not based on a specific observed slow case, just a reasonable ceiling.
- **`output_format` defaults to `"summary"`, not `"full"`.** Full Hayabusa findings carry `Details`/`ExtraFieldInfo`/`RuleID` that add little value for a first-pass triage and bloat the response; a caller who needs everything opts in with `output_format="full"`. This is a behavior change from the original `scan_evtx` (which always returned full findings) — anything built against the old default shape needs `output_format="full"` to keep working.
- **`rule_filter`/`max_results` implemented as Python post-processing, not new Hayabusa CLI flags.** Hayabusa has no "substring match on rule title" or "cap result count" option, so unlike `min_severity` these can't be pushed down into the `hayabusa` invocation — they filter/truncate the parsed JSONL after the fact.
- **`get_hayabusa_rules` parses rule YAML with regex line-scanning instead of adding a PyYAML dependency.** `requirements.txt` only lists `mcp`, matching the project's stdlib-only convention for the download scripts. Sigma/Hayabusa rule files have a consistent set of unindented top-level scalar fields (`title`, `id`, `level`, `status`, `tags`, `description`), so a line scan anchored to column 0 is sufficient for a listing/discovery tool — Hayabusa itself still does real YAML parsing when a rule is actually used to scan.
- **`get_hayabusa_rules` defaults `max_results` to 50, not unlimited.** The local rule set is ~4,961 files; returning all of them by default would flood an MCP response for little benefit when the caller is just browsing.
- **Claude Desktop config: absolute path in `args`, not `cwd`; and the real config file isn't where you'd expect.** This machine runs the MSIX-packaged Claude Desktop build (`Claude_pzs8sxrjxfjjc`), which Windows redirects to its own virtualized `%APPDATA%`. The result: `C:\Users\<user>\AppData\Roaming\Claude\claude_desktop_config.json` (the path every generic guide points to) exists but is inert — the packaged app never reads it. The file it actually reads/writes is `C:\Users\<user>\AppData\Local\Packages\Claude_pzs8sxrjxfjjc\LocalCache\Roaming\Claude\claude_desktop_config.json`. Discovered by editing the "obvious" path, restarting Desktop, and finding no `mcp-server-hayabusa.log` was ever created — then confirming via `Get-Process`/package inspection that the app's actual data directory was elsewhere. Separately, the app rewrites this config on every launch and normalizes each `mcpServers` entry down to just `command`/`args` — a `cwd` key set by hand gets silently dropped, which broke the first fix attempt (`python.exe: can't open file 'C:\\WINDOWS\\system32\\server.py'`, since `C:\WINDOWS\system32` is the app's default process cwd). Fixed by passing the absolute path to `server.py` as an `args` element instead of depending on `cwd` — safe because `HAYABUSA_DIR` in `server.py` is computed from `Path(__file__).resolve().parent`. Verified working via `logs\mcp-server-hayabusa.log` showing `tools/list` returning both tools with no disconnect.
- **Bug found and fixed: `_parse_rule_summary` stopped scanning at `detection:`, silently losing every field placed after it.** The original implementation assumed all metadata (`title`, `id`, `level`, `status`, `tags`, `description`) sat *above* the `detection:` block and `break`ed out of the line loop as soon as it saw a line starting with `detection:`. In practice a large share of Sigma rules put `level:` (and sometimes `tags:`) *after* `detection:`, near the end of the file (e.g. `hayabusa/rules/sigma/.../CreateDump Process Dump.yml` has `level: high` as its last line). This made every affected rule read back `level: null` — caught when a `get_hayabusa_rules(keyword="credential")` breakdown by MITRE technique showed all 120+ `T1003.001` rules with `level: null`, which was implausible for a well-maintained credential-dumping ruleset. Fix: removed the early `break` entirely and scan to EOF — safe because the field regexes (`_TITLE_RE`, `_ID_RE`, `_LEVEL_RE`, etc.) are anchored to column 0 (`^title:`, `^level:`, ...), so indented lines inside the `detection:` mapping never match them regardless of where the scan stops. Verified fix by re-running `get_hayabusa_rules` unfiltered: 0 of 4,961 rules have `level: null` post-fix, versus 100% pre-fix.

## Detection engineering knowledge base

A second layer built on the same server: a curated Sigma rule set and MITRE ATT&CK technique/coverage lookups, exposed mainly as MCP *resources* (browsable/discoverable, not invoked with arguments), plus one tool (`analyze_coverage`) for querying that same coverage data directly by technique or tactic.

### What we built

- **`./rules/`**: 24 curated Sigma detection rules (YAML), checked into git. 6 hand-authored (LSASS access/T1003.001, Kerberoasting/T1558.003, DCSync/T1003.006, Pass-the-Hash/T1550.002), plus 18 unmodified rules copied from upstream [SigmaHQ/sigma](https://github.com/SigmaHQ/sigma) (original `author`/`references` preserved, filenames prefixed `sigmahq_`) broadening coverage across credential-access, lateral-movement, and persistence techniques. Distinct from `./hayabusa/rules/` (the gitignored, downloaded-on-demand full checkout `get_hayabusa_rules` reads) — this is a deliberately curated cross-section, not a mirror.
- **`scripts/download_attack_data.py`**: downloads the MITRE ATT&CK Enterprise STIX bundle (~50MB) from a fixed URL (the [attack-stix-data](https://github.com/mitre-attack/attack-stix-data) repo's `master` branch) into `./attack/enterprise-attack.json`, stdlib `urllib` only, same fetch-on-demand/gitignored pattern as `download_hayabusa.py`/`download_sample_evtx.py`.
- **Four `detection://` MCP resources in `server.py`**, all registered via `@mcp.resource(...)`:
  - `detection://rules` — static resource; lists every rule in `./rules/` (title, id, level, status, tags, derived ATT&CK `techniques`, description) via `_sigma_rule_summary()`, which reuses `_parse_rule_summary()` (the same parser `get_hayabusa_rules` uses).
  - `detection://rules/{rule_name}` — template resource; returns one rule's raw YAML by filename stem, case-insensitive, extension optional. Raises `ValueError` (→ `ResourceError`) if no match.
  - `detection://rules/by-technique/{technique_id}` — template resource; lists rules whose derived `techniques` include the given (normalized) ID. No match returns `count: 0`, not an error.
  - `detection://attack/techniques/{technique_id}` — template resource; looks up the technique's name/description/URL from the downloaded ATT&CK data and cross-references it against `./rules/`, returning a `coverage` verdict (`covered`/`partial`/`gap`) alongside the matching rules. Raises if the ATT&CK data hasn't been downloaded or the ID doesn't exist.
- **`analyze_coverage(target: str) -> dict` tool in `server.py`**: the analytical counterpart to `detection://attack/techniques/{technique_id}` — a **tool**, not a resource, so it can take either a single technique ID or a whole tactic name in one call instead of requiring one lookup per technique.
  - `target` matching a technique-ID shape (`_TECHNIQUE_ID_RE`, optional `T` prefix) returns the same shape as the ATT&CK resource: name, tactics, `coverage` verdict, matching rules.
  - Otherwise `target` is treated as a tactic name (normalized via `_normalize_tactic_name()` to ATT&CK's shortname form, e.g. `"Credential Access"` → `"credential-access"`), resolved against a new `_load_attack_tactics()` cache (same STIX-bundle-parse-once-per-process pattern as `_load_attack_techniques()`), and every technique under that tactic (via each technique's `kill_chain_phases`, now also captured by `_load_attack_techniques()`) is bucketed into `covered`/`partial`/`gap` using the existing `_assess_technique_coverage()` — returning per-bucket counts and lists.
  - Returns `{"error": ...}` on failure (unknown technique/tactic, missing ATT&CK data, blank `target`) — the tool convention, not the resources' raise-and-wrap convention.
- **`.gitignore`**: added `attack/`, alongside the existing `hayabusa/`/`samples/*.evtx` entries.

### What's left to do

- **No automated test coverage.** `tests/test_scan_evtx.py` only covers the original two tools; the four `detection://` resources and `analyze_coverage` have only been exercised manually in-session, not via a repeatable test script.
- **`mappings/` not started.** The original plan called for explicit, hand-curated ATT&CK technique-to-rule mapping files, additive to what the resources already derive on the fly from each rule's `tags:`. Not created — current technique lookups are 100% tag-derived.
- **`./rules/` is curated, not comprehensive.** 24 rules across 4(+) tactics is a demonstration set, not full coverage of even the credential-access/lateral-movement/persistence tactics it targets, let alone all of ATT&CK. `detection://attack/techniques/{technique_id}` and `analyze_coverage`'s `gap`/`partial` verdicts will reflect that for the vast majority of ATT&CK's ~600 techniques and, for a tactic-level query, most of a given tactic's technique list.
- **ATT&CK data is a manual, unversioned download.** `scripts/download_attack_data.py` always pulls `master`, with no version pinning or staleness check; the server has no way to know if `./attack/enterprise-attack.json` is out of date relative to the currently published ATT&CK release.

### Decisions made and why

- **A full upstream Sigma mirror was considered and rejected for `./rules/`.** The user's first ask ("add all the Sigma rules") would have meant copying ~4,700 files (all platforms) into git — considered, but it would duplicate `./hayabusa/rules/` (already the full checkout, gitignored and fetched on demand), blow up repo size for a project documented as having a *curated* rule set, and — since `./rules/` is a flat directory, not split into `hayabusa`/`sigma` subdirectories like the upstream checkout — risk filename collisions in `_find_sigma_rule_file()`'s stem-only lookup. Resolved by asking the user, who chose a curated expansion across related tactics (credential-access, lateral-movement, persistence) instead.
- **MCP resources, not tools, for the knowledge base.** Rules and ATT&CK techniques are things to *browse/discover* by identifier (a rule name, a technique ID), which is what MCP resources (with URI templates) are for; `@mcp.tool()` was kept for the two actions that take open-ended parameters and *do* something (run a scan, search by keyword).
- **Resource errors raise instead of returning `{"error": ...}`.** That dict convention is specific to the two tools (documented in CLAUDE.md as "never lets a failure crash the server" via a return value). For resources, FastMCP already wraps any raised exception into a proper MCP `ResourceError` (confirmed by reading `mcp/server/fastmcp/server.py`'s `read_resource()`), so raising `ValueError`/`FileNotFoundError` for a not-found case is the idiomatic FastMCP pattern, not a gap in error handling.
- **`_sigma_rule_summary()` reuses `_parse_rule_summary()` rather than a second parser.** Both `get_hayabusa_rules` and the new resources need the same title/id/level/status/tags/description extraction from a Sigma YAML file; the only new pieces needed were `rule_name` (filename stem) and `techniques` (ATT&CK IDs regex-extracted from `tags:`), so those are layered on top instead of duplicating the line-scan logic.
- **The ATT&CK STIX bundle is parsed once and cached in memory (`_attack_techniques_cache`), not per-request.** It's ~50MB of JSON; re-parsing it on every `detection://attack/techniques/{...}` read would be wasteful for data that's static for the life of the server process. The tradeoff (documented) is that a fresher download requires restarting the server, not just re-running the download script.
- **`scripts/download_attack_data.py` uses one fixed URL, unlike the other two download scripts.** `download_hayabusa.py` and `download_sample_evtx.py` both resolve a moving target dynamically (a GitHub release asset, a specific file in a sample repo). The ATT&CK STIX bundle's location on `attack-stix-data`'s `master` branch is stable by convention — there's no versioned-release indirection to resolve, so a fixed URL is simpler and equally robust.
- **Coverage assessment includes parent/child technique relationships, not just exact tag matches.** A rule tagged for a parent technique (e.g. `attack.t1053`) plausibly catches some but not all of its sub-techniques' variants, and vice versa; treating that as `"partial"` rather than `"gap"` (exact match only) or `"covered"` (any relation counts fully) gives a more honest coverage signal than either extreme.
- **`analyze_coverage` is a tool, everything else knowledge-base-related is a resource.** The four `detection://` resources fit the "browse by identifier" pattern MCP resources are for (a rule name, a technique ID). A tactic-wide coverage sweep doesn't fit that pattern — there's no single URI to browse to, since it aggregates across dozens of techniques and returns a computed summary (counts, three buckets), which is squarely what `@mcp.tool()` is for. Implementing it as a tool also gets it the `{"error": ...}` convention instead of a raised exception, consistent with `scan_evtx`/`get_hayabusa_rules`.
- **One `target` parameter, not separate `technique_id`/`tactic` parameters.** Auto-detecting via `_TECHNIQUE_ID_RE` (does the input look like `T####[.###]`, `T` optional?) keeps the tool's interface to a single string a caller can pass without first deciding which lookup mode they want — the mode is implied by the shape of what they already have on hand (a technique ID from a rule tag, or a tactic name from ATT&CK's own taxonomy).
- **Tactic-mode coverage reuses `_assess_technique_coverage()` per technique rather than a separate aggregate algorithm.** Every technique in the tactic gets the same covered/partial/gap logic as a single-technique lookup, just run in a loop and bucketed — so the tactic-level verdicts stay consistent with what a caller would get by looking up each technique individually via `detection://attack/techniques/{technique_id}`.
